# Scheduler Lab - CS 3502 Project 2

A web-based CPU scheduling simulator built for CS 3502 (Operating Systems)
at Kennesaw State University, using the **hybrid approach**: the scheduling
algorithm logic from the assignment was implemented as a standalone
JavaScript engine with a custom browser interface, instead of extending
the provided C# starter.

## Live Demo
https://olamide-aa.github.io/CS3502/project2/

## Algorithms

Starter set, reimplemented: **FCFS**, **SJF** (non-preemptive),
**Priority** (non-preemptive, lower number = higher priority),
**Round Robin** (configurable quantum).

New algorithms added for this project:

- **SRTF** (Shortest Remaining Time First) - the preemptive form of SJF.
  At every time unit the scheduler runs whichever arrived process has the
  least remaining work, preempting the current process if a shorter one
  arrives.
- **HRRN** (Highest Response Ratio Next) - non-preemptive. At each
  scheduling decision it selects the ready process with the highest
  response ratio (waiting + burst) / burst. The ratio rises the longer a
  process waits, so long jobs cannot starve. HRRN exists to fix the
  starvation problem that SJF and SRTF share.

## Metrics

For every algorithm on the same workload: average waiting time, average
turnaround time, average response time, CPU utilization, and throughput.
Results display in a side-by-side comparison table with the best value
per metric highlighted, plus a Gantt timeline and per-process table for
each algorithm.

## Running it

No build step and no dependencies. Open `index.html` in any modern
browser. The page loads with the assignment's example workload
(P1-P4) so the first run can be verified by hand.

## Reproducing report data

The workload generator uses a seeded deterministic RNG (mulberry32), so
a given (count, type, seed) always produces the identical process set.
All data in the report can be regenerated by entering the seed listed
with each table.

Workload types: CPU-bound (bursts 15-40), I/O-bound (bursts 1-6),
mixed (70% short, 30% long).

## Testing

`engine.js` contains a test harness that runs under Node
(`node engine.js`). It checks all six algorithms against hand-calculated
results for the assignment's example workload, verifies schedule
conservation (total scheduled time equals total burst time), and smoke
tests a 150-process workload through every algorithm.

## Files

- `index.html` - page structure and styling
- `engine.js` - pure scheduling logic, no UI dependencies (the extracted
  algorithm core of the hybrid approach)
- `ui.js` - table input, workload generator controls, comparison
  rendering, Gantt timelines, CSV export
